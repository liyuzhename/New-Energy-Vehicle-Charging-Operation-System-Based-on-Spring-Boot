# 前端问题描述

以下问题经后端代码排查，确认为前端实现问题，需前端修复。

---

## 1. 运营商端充电桩管理 - 新建充电桩报"桩编号不能为空; 桩类型不能为空"

**现象：**
填写完所有字段（所属充电站、桩编号、功率）后点击创建，报错：
```
Uncaught (in promise) Error: 桩编号不能为空; 桩类型不能为空
```

**原因分析：**
后端 `PileCreateRequest` 要求的字段名为：
- `pileNo`（桩编号）
- `stationId`（所属充电站 ID，Long 类型）
- `pileType`（桩类型，必填）
- `power`（功率，选填）
- `status`（初始状态，选填，不传默认 `IDLE`）

前端提交时字段名与后端不匹配（如用了 `pile_no`、`pile_type` 等下划线命名），或者**桩类型字段未绑定到表单**（用户填写了但前端代码未将其加入请求体）。

**修复建议：**
1. 检查 `OperatorPile.vue` 中 `handleSavePile` 方法构造请求体时，确保字段名使用驼峰命名：
   ```js
   const params = {
     pileNo: form.pileNo,
     stationId: form.stationId,
     pileType: form.pileType,   // 确保此字段有值且字段名正确
     power: form.power,
     status: form.status        // 可选，不传默认 IDLE
   }
   ```
2. 确认表单中"桩类型"选择框已正确双向绑定到 `form.pileType`。

## 2. 运营商端计费规则管理 - 新建规则报"开始小时不能为空; 结束小时不能为空; 生效日期不能为空"

**现象：**
填写完所有字段后点击新建，报错：
```
Uncaught (in promise) Error: 开始小时不能为空; 结束小时不能为空; 生效日期不能为空
```

**原因分析：**
后端 `BillingRuleCreateRequest` 要求的字段名为：
- `stationId`（充电站 ID，Long）
- `periodType`（时段类型，如 `PEAK`/`FLAT`/`VALLEY`）
- `startHour`（开始小时，Integer，0-23）
- `endHour`（结束小时，Integer，1-24）
- `electricityPrice`（电费单价，BigDecimal）
- `servicePrice`（服务费单价，BigDecimal）
- `effectiveDate`（生效日期，格式 `yyyy-MM-dd`）

前端提交时字段名不匹配，或时间选择器返回的是完整日期时间字符串而非 `yyyy-MM-dd` 格式，导致后端反序列化 `LocalDate` 失败，字段值为 null。

**修复建议：**
1. 检查 `OperatorBilling.vue` 中 `handleSave` 方法，确保字段名驼峰命名且类型正确：
   ```js
   const params = {
     stationId: form.stationId,
     periodType: form.periodType,
     startHour: Number(form.startHour),       // 确保是数字类型
     endHour: Number(form.endHour),           // 确保是数字类型
     electricityPrice: form.electricityPrice,
     servicePrice: form.servicePrice,
     effectiveDate: form.effectiveDate        // 格式必须为 "yyyy-MM-dd"，如 "2024-01-01"
   }
   ```
2. 若使用 Element Plus 的 `DatePicker`，需用 `value-format="YYYY-MM-DD"` 属性确保输出格式正确。

## 3. 运营商端计费规则管理 - 编辑时开始/结束时间不变

**现象：**
编辑计费规则时，修改开始时间和结束时间后提交，显示"编辑成功"，但时间字段未更新，只有价格发生了变化。

**原因分析：**
后端 `BillingRuleUpdateRequest` 的 `startHour` 和 `endHour` 字段类型为 `Integer`，采用判空更新（`if (request.getStartHour() != null)`）。后端逻辑本身正确。

问题在于前端编辑表单提交时，`startHour` 和 `endHour` 字段**未包含在请求体中**，或者传的是字符串类型（如 `"8"`）而非数字（`8`），导致后端接收到 null 或反序列化失败，跳过了时间字段的更新。

**修复建议：**
1. 检查 `OperatorBilling.vue` 编辑提交逻辑，确保 `startHour` 和 `endHour` 包含在 PUT 请求体中，且为数字类型：
   ```js
   const params = {
     periodType: form.periodType,
     startHour: Number(form.startHour),   // 必须是 Number，不能是字符串
     endHour: Number(form.endHour),       // 必须是 Number，不能是字符串
     electricityPrice: form.electricityPrice,
     servicePrice: form.servicePrice,
     effectiveDate: form.effectiveDate    // 格式 "yyyy-MM-dd"
   }
   ```
2. 若时间选择器绑定的是字符串，需在提交前做类型转换。

## 4. 记住密码功能 - 勾选后退出登录仍需重新输入

**现象：**
登录时勾选"记住密码"，退出登录后重新打开页面，仍需手动输入账号密码。

**原因分析：**
后端已支持 `rememberMe` 字段（`true` 时生成 7 天有效期 token，`false` 时 1 天），但前端需要配合实现本地持久化：
1. 登录时将 `rememberMe: true` 传给后端
2. 勾选记住密码时，将账号密码保存到 `localStorage`
3. 页面加载时从 `localStorage` 读取并回填表单

**修复建议：**
```js
// 登录时
const params = {
  username: form.username,
  password: form.password,
  rememberMe: form.rememberMe
}

// 登录成功后，若勾选了记住密码
if (form.rememberMe) {
  localStorage.setItem('rememberedUsername', form.username)
  localStorage.setItem('rememberedPassword', form.password)
} else {
  localStorage.removeItem('rememberedUsername')
  localStorage.removeItem('rememberedPassword')
}

// 页面 mounted 时回填
const rememberedUsername = localStorage.getItem('rememberedUsername')
const rememberedPassword = localStorage.getItem('rememberedPassword')
if (rememberedUsername) {
  form.username = rememberedUsername
  form.password = rememberedPassword
  form.rememberMe = true
}
```
