# 前端问题说明

以下问题经后端代码分析确认为**纯前端渲染逻辑问题**，后端返回数据完全正确，需前端修复。

---

## 问题1：管理员端用户管理界面所有用户状态显示为"禁用"

**现象：** 管理员端用户管理界面中，状态列的 toggle 开关全部显示为关闭（灰色/禁用），与实际状态不符。后端接口返回所有启用用户的 `status` 均为 `1`，经 curl 验证正确。

**后端返回数据：**
- 接口：`GET /api/admin/user/list`
- `status` 字段类型：`Integer`，`1` = 启用，`0` = 禁用

**问题原因：** Element Plus 的 `<el-switch>` 组件 `active-value` 和 `inactive-value` 设置反了，导致 `1` 被识别为"关闭"状态。

**修复建议：** 找到用户列表中的 switch 组件，修正 `active-value`：
```vue
<!-- 错误写法（active-value 设成了 0） -->
<el-switch v-model="row.status" :active-value="0" :inactive-value="1" />

<!-- 正确写法 -->
<el-switch v-model="row.status" :active-value="1" :inactive-value="0" />
```

同时检查切换状态时的确认弹窗逻辑，确保：
```js
// 正确逻辑
row.status === 1  →  弹窗显示"确定禁用 xxx 吗？"，调用接口传 status=0
row.status === 0  →  弹窗显示"确定启用 xxx 吗？"，调用接口传 status=1
```

## 问题2：管理员端运营商管理所有运营商状态显示为"已禁用"

**现象：** 管理员端运营商管理页面中，所有运营商的状态都显示为"已禁用"，与实际状态不符。

**后端返回数据：**
- 接口：`GET /api/admin/operator/list`
- `status` 字段类型：`Integer`
- 值含义：`1` = 启用，`0` = 禁用（与用户表相同，运营商本质上是 `user` 表中 `role = 'OPERATOR'` 的记录）

**问题原因：** 与问题1相同，前端渲染运营商状态时 `0/1` 判断逻辑反了。

**修复建议：** 检查运营商列表组件中 `status` 的判断逻辑，与用户管理保持一致：
```js
status === 1  →  显示"启用"
status === 0  →  显示"禁用"
```

## 问题3：运营商端新建/编辑充电站时联系电话和营业时间不保存

**现象：** 新建或编辑充电站时填写了联系电话和营业时间，保存后数据丢失（描述字段可以正常保存）。

**后端验证：** 经 curl 测试确认，后端接口 `PUT /api/operator/station/{id}` 完全正常，传入 `contactPhone` 和 `businessHours` 后数据库正确保存。

**后端接口字段（均已支持）：**
```json
{
  "contactPhone": "13900001234",
  "businessHours": "08:00-22:00",
  "description": "站点描述"
}
```

**问题原因：** 前端表单提交时，`contactPhone` 和 `businessHours` 字段没有被包含在请求体中。可能原因：
- 表单数据对象中字段名与后端不一致（如前端用 `contact_phone` 而后端期望 `contactPhone`）
- 表单绑定遗漏了这两个字段
- 提交前数据被过滤掉了

**修复建议：** 检查充电站新建/编辑表单的提交逻辑，确保请求体中包含 `contactPhone`（驼峰命名）和 `businessHours` 字段。

## 问题4：运营商端充电桩列表桩编号列为空白

**现象：** 充电桩管理列表中"桩编号"列显示为空白（所属充电站列正常显示）。

**后端验证：** 经 curl 测试确认，`GET /api/operator/pile/list` 接口返回数据中 `pileNo` 字段有值，例如：
```json
{ "id": 3, "pileNo": "PILE-WH-001", "stationName": "绿城西湖充电中心", ... }
```

**问题原因：** 前端表格列绑定的字段名与后端返回的字段名不一致。后端返回的是驼峰命名 `pileNo`，前端可能绑定的是 `pile_no`（下划线命名）或其他错误字段名。

**修复建议：** 检查充电桩列表组件的表格列定义，将桩编号列的字段绑定改为 `pileNo`：
```js
// 错误（可能的写法）
{ prop: 'pile_no', label: '桩编号' }

// 正确
{ prop: 'pileNo', label: '桩编号' }
```

## 问题6：运营商端充电枪管理添加充电枪时报"枪类型不能为空"

**现象：** 充电枪管理弹窗中填写"接口类型"和"功率"后点击添加，报错"枪类型不能为空; 枪口编号不能为空"。

**后端接口 `POST /api/operator/gun` 期望的请求体：**
```json
{
  "pileId": 3,
  "gunNo": "A1",
  "gunType": "GB_DC",
  "power": 120
}
```

**需要前端修改两处：**

1. 表单中加回"枪口编号"输入框（必填），由用户手动填写与实物对应的编号，后端会校验同一充电桩下编号唯一

2. "接口类型"下拉框提交时字段名必须是 `gunType`：
```js
// 错误（可能的写法）
{ connectorType: form.interfaceType }

// 正确
{ pileId: props.pileId, gunNo: form.gunNo, gunType: form.gunType, power: form.power }
```

## 问题5：运营商端计费规则时间选择器无法设置结束时间为 24:00

**现象：** 新增计费规则时，时间选择器只能选到 `00:00`，无法选 `24:00`。当开始时间为 `18:00`、结束时间选 `00:00` 时，前端自身校验报"结束时间必须晚于开始时间"，导致无法提交。

**后端说明：** 后端已修复，`endHour=0` 会自动转换为 `24`（表示当天 24:00），可以正常创建 18:00-24:00 的规则。但前端的校验逻辑会在提交前拦截，请求根本到不了后端。

**问题原因：** 前端在提交前校验 `endHour > startHour`，而时间选择器选 `00:00` 对应的值是 `0`，`0 < 18` 所以被拦截。

**修复建议：** 修改前端的时间校验逻辑，将 `endHour=0` 视为 `24`：

```js
// 错误写法
if (form.endHour <= form.startHour) {
  ElMessage.error('结束时间必须晚于开始时间')
  return
}

// 正确写法：endHour=0 代表 24:00
const endHour = form.endHour === 0 ? 24 : form.endHour
if (endHour <= form.startHour) {
  ElMessage.error('结束时间必须晚于开始时间')
  return
}
```

同时建议在时间选择器的选项中增加 `24:00` 选项（值为 `0` 或 `24`），或在 `00:00` 旁边加备注说明"00:00 表示当天结束（24:00）"，避免用户困惑。

## 问题7：用户端充电站列表所有充电站状态显示为"已关闭"

**现象：** 用户端充电站列表中，所有充电站右上角均显示"已关闭"标签，与实际状态不符。

**后端验证：** 经 curl 测试确认，`GET /api/station/list` 接口返回的所有充电站 `status` 均为 `"ONLINE"`，后端数据完全正确。

**问题原因：** 前端渲染充电站状态标签时，`ONLINE` 和 `OFFLINE` 的判断逻辑写反了。

**修复建议：** 找到充电站卡片组件中的状态标签渲染逻辑，修正判断条件：
```js
// 错误写法
station.status === 'ONLINE'  →  显示"已关闭"
station.status === 'OFFLINE' →  显示"营业中"

// 正确写法
station.status === 'ONLINE'  →  显示"营业中"（绿色）
station.status === 'OFFLINE' →  显示"已关闭"（灰色）
```
