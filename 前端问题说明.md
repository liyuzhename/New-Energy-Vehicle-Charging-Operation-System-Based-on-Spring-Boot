# 前端问题说明

本文档记录后端修复后，前端需要对应调整的内容。

---

## 问题1：用户端钱包流水按类型搜索无反应

**后端修改：** `GET /api/user/wallet/records` 新增可选查询参数 `type`。

**前端需要修改：**

调用流水记录接口时，将用户选择的类型作为 `type` 参数传入：

```js
// 示例
GET /api/user/wallet/records?type=RECHARGE&page=1&size=10
GET /api/user/wallet/records?type=CONSUME&page=1&size=10
GET /api/user/wallet/records?page=1&size=10  // 不传 type 则返回全部
```

`type` 可选值：
- `RECHARGE`：充值
- `CONSUME`：消费
- `REFUND`：退款

## 问题2：用户端每隔一段时间弹出"当前没有进行中的充电订单"

**后端修改：** `GET /api/order/charging` 在无进行中订单时不再抛异常，改为返回 `{"code":200,"data":null}`。

**前端需要修改：**

轮询该接口时，判断 `data` 是否为 `null`，为 `null` 时静默处理，不弹出错误提示：

```js
// 修改前（错误做法）
if (res.code !== 200) {
  showToast(res.message) // 会弹出"当前没有进行中的充电订单"
}

// 修改后（正确做法）
if (res.code === 200 && res.data !== null) {
  // 更新充电状态
} else {
  // 无充电订单，静默处理，不弹窗
}
```

## 问题3：用户端订单列表字段为空（充电站、充电时长、充电量、金额）

**后端修改：** `GET /api/order/my` 返回的每条 `OrderVO` 新增以下字段：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `stationName` | String | 充电站名称 |
| `pileNo` | String | 充电桩编号 |
| `power` | BigDecimal | 充电桩功率（kW） |
| `plateNo` | String | 车牌号 |
| `chargeDuration` | Long | 充电时长（秒），已完成订单为实际时长，充电中为当前已充时长 |

**前端需要修改：**

订单列表直接使用上述新字段渲染，无需前端自行关联查询：

```js
// 充电站名称
item.stationName

// 充电时长（秒转分钟/小时显示）
const minutes = Math.floor(item.chargeDuration / 60)

// 充电量
item.chargeKwh

// 金额
item.totalFee

// 车牌号
item.plateNo
```

## 问题4：充电页面充电桩/充电站/实时功率为空，已充电量和预估费用不更新

**后端修改：** `GET /api/order/charging` 返回的 `OrderVO` 新增以下字段（实时计算）：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `stationName` | String | 充电站名称 |
| `pileNo` | String | 充电桩编号 |
| `power` | BigDecimal | 充电桩功率（kW） |
| `plateNo` | String | 车牌号 |
| `chargingSeconds` | Long | 已充电时长（秒） |
| `chargeKwh` | BigDecimal | 已充电量（kWh），实时按功率×时长计算 |
| `estimatedFee` | BigDecimal | 预估费用（元），按计费规则实时计算 |
| `chargeDuration` | Long | 同 `chargingSeconds` |

**前端需要修改：**

充电页面轮询 `GET /api/order/charging`，直接使用返回的字段渲染，无需前端自行计算：

```js
// 充电站
page.stationName = res.data.stationName

// 充电桩
page.pileNo = res.data.pileNo

// 实时功率
page.power = res.data.power + ' kW'

// 已充电量
page.chargeKwh = res.data.chargeKwh + ' kWh'

// 预估费用
page.estimatedFee = res.data.estimatedFee + ' 元'

// 充电时长
page.duration = formatSeconds(res.data.chargingSeconds)
```

## 问题5：用户端订单详情合计金额一直为0

**后端修改：**
1. `GET /api/order/{orderId}` 返回的 `OrderDetailVO` 新增字段：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `stationName` | String | 充电站名称 |
| `pileNo` | String | 充电桩编号 |
| `plateNo` | String | 车牌号 |
| `chargeDuration` | Long | 充电时长（秒） |

2. 已完成订单的 `feeDetail` 直接使用数据库存储值，不再重算：

| 字段名 | 说明 |
|--------|------|
| `feeDetail.totalChargeKwh` | 充电量（kWh） |
| `feeDetail.totalChargeFee` | 电费（元） |
| `feeDetail.totalServiceFee` | 服务费（元） |
| `feeDetail.totalFee` | 合计（元） |

3. 充电中（`status=CHARGING`）订单实时计算预估费用并填充 `totalFee`。

**前端需要修改：**

订单详情页直接使用以下字段渲染，无需前端自行关联查询：

```js
// 充电站
detail.stationName

// 充电桩
detail.pileNo

// 车牌号
detail.plateNo

// 充电时长（秒转分钟显示）
const minutes = Math.floor(detail.chargeDuration / 60)

// 电费、服务费、合计 —— 统一从 feeDetail 读取
detail.feeDetail.totalChargeFee    // 电费
detail.feeDetail.totalServiceFee   // 服务费
detail.feeDetail.totalFee          // 合计
```

## 问题6：运营商端订单管理字段为空，按充电站搜索无效

**后端修改：**

1. `GET /api/operator/order/list` 新增可选查询参数 `stationId`，用于按充电站过滤订单
2. 返回的每条 `OrderVO` 补充了以下关联字段（与用户端订单列表一致）：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `stationName` | String | 充电站名称 |
| `pileNo` | String | 充电桩编号 |
| `plateNo` | String | 车牌号（用户） |
| `chargeDuration` | Long | 充电时长（秒） |
| `userName` | String | 用户昵称（无昵称时为用户名） |

**前端需要修改：**

1. 充电站搜索下拉框选中后，将充电站的 `id` 作为 `stationId` 参数传给接口：

```js
// 示例
GET /api/operator/order/list?stationId=5&page=1&size=10

// 不传 stationId 则返回该运营商名下全部订单
GET /api/operator/order/list?page=1&size=10
```

2. 订单列表直接使用返回的新字段渲染，无需前端自行关联：

```js
// 用户（昵称/用户名）
item.userName

// 充电站
item.stationName

// 充电桩
item.pileNo

// 时长（秒转分钟）
const minutes = Math.floor(item.chargeDuration / 60)

// 充电量
item.chargeKwh  // 单位 kWh

// 金额
item.totalFee
```
