```js

meta: { roles: ['ADMIN'] }

// 修改后
meta: { roles: ['OPERATOR'] }
```

## 用户端充电预约 — 时间跨天问题

**后端支持跨天预约，问题在前端构造时间的逻辑。**

后端接口接收的是完整的 `LocalDateTime`（日期+时间），例如：
```json
{
  "pileId": 1,
  "startTime": "2026-03-27T22:00:00",
  "endTime": "2026-03-28T00:00:00"
}
```

**前端修改点：** 在发起预约弹窗中，构造 `endTime` 时需要判断是否跨天：
- 如果结束时间（时分）> 开始时间（时分）：`endTime` 日期 = 预约日期
- 如果结束时间（时分）≤ 开始时间（时分）：`endTime` 日期 = 预约日期 + 1天

同时，前端的校验逻辑也需要去掉"结束时间必须晚于开始时间"的简单时间比较，改为比较完整的 `DateTime`（跨天时结束时间的时分虽然小，但日期+1后整体仍然晚于开始时间）。

## 预约功能改造 — 粒度从充电桩升级为充电枪

### 改造背景

原预约按充电桩粒度，同一充电桩同一时段只能有一个预约。改造后按**充电枪**粒度，同一充电桩下不同充电枪可以被不同用户同时预约，且同一把枪在不同时段也可以被多次预约。

---

### 1. 查询可预约充电枪（新增接口）

**用途：** 预约弹窗中，用户选好时间段后，调用此接口获取该时段内可预约的枪列表供用户选择。

```
GET /api/station/{stationId}/guns-available?startTime=&endTime=
```

**参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| stationId | Long（路径参数） | 充电站ID |
| startTime | ISO 8601 datetime | 预约开始时间，如 `2026-03-28T10:00:00` |
| endTime | ISO 8601 datetime | 预约结束时间，如 `2026-03-28T12:00:00` |

**响应示例：**
```json
{
  "code": 200,
  "data": [
    { "id": 3, "pileId": 3, "gunNo": "A", "gunType": "DC", "power": 120.0, "status": "IDLE" },
    { "id": 4, "pileId": 3, "gunNo": "B", "gunType": "DC", "power": 120.0, "status": "IDLE" }
  ]
}
```

**前端修改点：**
- 预约弹窗新增"选择充电枪"步骤：用户选完时间段后调用此接口，展示可选枪列表（显示 `gunNo`、`gunType`、`power`）
- 用户选中某把枪后，将 `gunId` 和 `pileId` 一起带入创建预约请求

---

### 2. 创建预约（接口变更）


**请求体变更：新增必填字段 `gunId`**

```json
{
  "pileId": 3,
  "gunId": 3,
  "startTime": "2026-03-28T10:00:00",
  "endTime": "2026-03-28T12:00:00"
}
```

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|----------|------|
| pileId | Long | 是 | 充电桩ID |
| gunId | Long | 是（新增） | 充电枪ID，必须属于该充电桩 |
| startTime | LocalDateTime | 是 | 预约开始时间 |
| endTime | LocalDateTime | 是 | 预约结束时间 |

**前端修改点：** 创建预约时必须传 `gunId`，否则后端返回400。

### 4. 确认到场（新增接口）

**用途：** 用户到达充电站后，在预约时间段内点击"确认到场"，将预约状态从 `PENDING` 改为 `CONFIRMED`。

```
PUT /api/reservation/{id}/confirm
```

**约束：**
- 当前时间必须在预约时间段 `[startTime, endTime)` 内
- 预约状态必须为 `PENDING`

**响应：**
```json
{ "code": 200, "message": "确认到场成功" }
```

**前端修改点：** 预约详情页/列表页，当预约状态为 `PENDING` 且当前时间已进入预约时间段时，显示"确认到场"按钮，点击调用此接口。

---

### 5. 预约转充电（新增接口）

**用途：** 用户确认到场后，点击"开始充电"，直接使用预约的充电枪创建充电订单。

```
POST /api/reservation/{id}/start-charging
```

**请求体（可选）：**
```json
{ "vehicleId": 1 }
```

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|----------|------|
| vehicleId | Long | 否 | 车辆ID |

**约束：** 预约状态必须为 `CONFIRMED`。

**响应：**
```json
{ "code": 200, "message": "充电已开始", "data": 9 }
```

`data` 为创建的充电订单ID，前端可用此ID跳转到充电中页面。

**前端修改点：** 预约详情页，当预约状态为 `CONFIRMED` 时，显示"开始充电"按钮，点击调用此接口，成功后跳转到充电中页面（传入返回的订单ID）。

---

### 6. 预约状态说明（完整版）

| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| PENDING | 已预约，等待到场 | 取消（开始前30分钟前）、确认到场（进入时间段后） |
| CONFIRMED | 已确认到场 | 开始充电 |
| CANCELLED | 已取消 | 无 |
| EXPIRED | 已过期（超时未到场或未充电） | 无 |

---

### 7. 充电枪状态说明（运营商端）

充电枪新增 `RESERVED` 状态，运营商枪列表中会出现此状态：

| 状态 | 说明 |
|------|------|
| IDLE | 空闲，可充电/可预约 |
| RESERVED | 已被预约（当前时间在预约时间段内） |
| CHARGING | 充电中 |
| FAULT | 故障 |

**前端修改点（运营商端）：** 充电枪状态列表中增加 `RESERVED` 状态的展示（如显示为"已预约"，用不同颜色标识）。
