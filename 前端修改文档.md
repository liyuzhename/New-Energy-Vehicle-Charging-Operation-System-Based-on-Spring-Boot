## 问题2 — 用户管理按状态搜索无效

**后端接口正常，问题在前端传参。**

接口：`GET /api/admin/user/list`

状态参数 `status` 支持以下值（传字符串）：
- `"0"` 或 `"disabled"` 或 `"禁用"` → 禁用用户
- `"1"` 或 `"enabled"` 或 `"启用"` → 启用用户

**前端修改点：** 确保搜索时将状态下拉框的值作为 `status` 参数传入请求，例如：
```
GET /api/admin/user/list?status=1&page=1&size=10
```

## 问题3 — 订单管理字段为空

**后端已修复，前端无需修改。**

接口 `GET /api/admin/order/list` 现在返回的每条记录包含以下新增字段：
- `userName`：用户昵称/用户名
- `stationName`：充电站名称

前端直接使用这些字段渲染即可。

## 问题4 — 故障管理字段为空 + 按充电站搜索无效

**后端已修复，前端需要适配新字段和新参数。**

接口：`GET /api/admin/fault/list`

**新增响应字段：**
- `operatorName`：运营商名称（原来为空）
- `reporterName`：提报人名称（原来为空）

**新增请求参数：**
- `stationId`（Long，可选）：按充电站 ID 过滤

示例：
```
GET /api/admin/fault/list?stationId=5&page=1&size=10
```

**前端修改点：**
1. 故障列表表格中"充电桩"列使用 `pileNo`，"运营商"列使用 `operatorName`，"提报人"列使用 `reporterName`
2. 按充电站搜索时，将选中的充电站 ID 作为 `stationId` 参数传入

## 问题6 — 公告管理搜索无效 + 无搜索按钮

**后端已修复，前端需要适配新参数并添加搜索按钮。**

接口：`GET /api/admin/announcement/list`

**新增请求参数：**
- `keyword`（String，可选）：按标题/内容模糊搜索
- `type`（String，可选）：按类型精确过滤，可选值：`NOTICE`（通知）、`MAINTENANCE`（维护公告）等
- `status`（String，可选）：按状态精确过滤，可选值：`ONLINE`（已上线）、`OFFLINE`（已下线）

示例：
```
GET /api/admin/announcement/list?type=NOTICE&status=ONLINE&page=1&size=10
```

**前端修改点：**
1. 在公告管理页面添加搜索区域，包含：
   - 标题关键词输入框（对应 `keyword` 参数）
   - 类型下拉框（对应 `type` 参数，选项：全部/NOTICE/MAINTENANCE）
   - 状态下拉框（对应 `status` 参数，选项：全部/ONLINE/OFFLINE）
   - 搜索按钮（点击后携带以上参数发起请求）

## 问题8 — 操作日志字段为空

**后端已修复，前端需要适配新字段。**

接口：`GET /api/admin/log/list`

**新增响应字段：**
- `role`：操作人角色（如 `ADMIN`、`OPERATOR`）

**现有字段说明（前端字段名对应）：**
- `operatorName`：操作人用户名
- `role`：角色
- `operation`：操作内容（格式：`HTTP方法 接口路径`，如 `GET /api/admin/user/list`）
- `method`：操作模块（格式：`控制器方法名`，如 `AdminController.listUsers(..)`）
- `createTime`：操作时间
- `status`：操作结果（`1`=成功，`0`=失败）

**前端修改点：**
1. "操作人"列使用 `operatorName`
2. "角色"列使用 `role`
3. "操作内容"列使用 `operation`
4. "操作模块"列使用 `method`
